# 服务边界检查清单

## 适用场景

- 新增或拆分 services 下的领域服务
- 出现跨服务强耦合、职责不清

## 前置条件

- 已有业务流程图或核心用例
- 已识别主实体：Course、Assignment、Stage、Group、MiniTask、Submission、Review、Grade

## 执行步骤

1. 确认单一职责：每个服务只拥有一个核心业务能力。
2. 确认数据所有权：每张核心表仅归一个服务主导。
3. 确认接口边界：跨服务只能通过明确 API 或事件交互。
4. 确认失败策略：外部调用必须定义超时、重试、降级。
5. 确认演进路径：拆分后是否支持灰度和回滚。

## 检查清单

- [ ] 服务命名与业务能力一致
- [ ] 不存在直接跨服务读写对方私有表
- [ ] 跨服务依赖有契约文档
- [ ] 关键链路有超时和重试策略
- [ ] 变更影响范围可追踪

## 常见反例

- 把多个业务域塞进一个超级服务
- 通过共享数据库表规避 API 设计
- 没有降级策略导致级联故障
