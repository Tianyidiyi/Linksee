# Linksee 课程场景与需求整理（V3）

## 一、范围说明

本版本只整理场景和需求，不展开技术方案、系统架构、数据库设计、性能优化。

明确四件事：

- 账号体系与权限边界。
- 助教子账号与批改职责。
- 批改和成绩变更的留痕规则。
- Task-MiniTask 两层任务机制。

## 二、场景定位

Linksee 在本阶段定位为“软件工程课程项目协作平台”，核心是把课程组织、项目发布、小组协作、过程记录和教学评价放到同一套流程中。

关键边界：

- 以课程为上层单位。
- 课程下包含班级这一层，用于明确学生归属与组队范围。
- 每门课程下由老师发布项目。
- 学生按项目组建小组并在小组内协作。
- 协作流程整体沿用原有团队协作逻辑（任务、沟通、文件、进度）。
- 文件管理中的 RAG 能力本阶段不实现。

非目标（本期不承诺）：

- 教务系统深度对接（统一身份、排课系统直连等）。
- 企业级复杂审批流与多校区多租户治理。
- 智能自动评审闭环与复杂 AI 分析能力。

## 三、角色与职责

### 1. 教务处

- 创建和维护课程基础信息。
- 指派课程负责人（老师）并导入或维护班级学生名单。
- 查看课程整体执行情况（进度、提交、批改覆盖）。
- 进行课程级管理与监督。

### 2. 老师

- 在课程下发布项目。
- 配置项目周期、阶段要求和提交要求。
- 查看各小组进度与提交情况。
- 对小组成果进行批改、反馈和评分。
- 创建助教子账号并分配课程范围。
- 对最终成绩进行确认发布。

### 3. 助教

- 由老师创建并归属于具体课程的子账号。
- 协助老师查阅提交材料。
- 执行批改、登记成绩、标注问题、跟踪修改。
- 协助维护课程项目过程记录。

助教权限说明：

- 默认拥有与老师接近的执行权限（查阅、批改、登记）。
- 最终成绩发布权保留给老师（避免责任边界不清）。

### 4. 学生

- 参与不同课程。
- 在不同课程中加入不同项目小组。
- 在所属小组中开展任务协作、沟通和材料提交。

## 四、班级与组队规则（新增）

### 1. 班级层的定位

- 课程下需要有班级这一层，用于管理本课程参与学生。
- 班级层的核心作用是限定项目组队范围，而不是单独做复杂教务管理。
- 课程、班级、项目、小组之间的主线为：课程 -> 班级 -> 项目 -> 小组。

### 2. 班级成员来源

- 教务处在课程创建后导入或维护本课程学生名单。
- 老师和助教基于班级名单开展项目管理和组队管理。
- 仅班级成员可参与本课程项目的小组组建与调整。

### 3. 学生身份标识

- 系统内部使用唯一用户标识管理账号关系。
- 学号作为学生业务字段保留，用于导入、展示和校验。
- 同一学生可以参与多门课程，但在同一项目中只能属于一个小组。

## 五、账号与权限规则（新增）

### 1. 账号类型

- 教务处账号：课程级管理与监管。
- 老师账号：课程负责人账号。
- 助教子账号：由老师创建，绑定课程范围。
- 学生账号：可跨课程参与多个项目小组。

### 2. 权限主张

- 教务处：查看课程整体进度、批改覆盖和审计记录。
- 老师：发布项目、管理任务框架、批改评分、最终成绩确认。
- 助教：执行查阅、批改、登记、反馈，但不做最终成绩发布。
- 学生：参与协作、执行任务、提交材料、查看反馈。

### 3. 关键权限约束

- 任何成绩修改必须留痕，所有相关操作可被老师和教务处审查。
- 同一学生可在不同课程有不同小组身份。
- 助教的执行操作必须纳入审计范围。

## 六、核心业务主线

1. 教务处设置课程。
2. 教务处导入课程班级名单并指定老师。
3. 老师在课程内发布项目。
4. 学生在项目下通过组队面板创建、加入或调整小组。
5. 小组获得独立任务空间，开展协作。
6. 小组进行课程群与小组群沟通。
7. 小组提交成果与过程文件（含仓库字段）。
8. 老师/助教查阅、批改、给出反馈。
9. 教务处与老师查看项目推进与批改进度。

## 七、核心场景

### 场景 A：课程创建与项目发布（Course）

- 教务处创建课程并完成课程基础配置。
- 教务处为课程导入班级学生名单。
- 老师在课程中创建项目并发布给学生。

老师在创建课程项目时，需要配置组队规则：

- 组队开始时间与截止时间。
- 每组最少人数与最多人数。
- 项目允许创建的最大小组数量。
- 组号分配方式与截止后的整理方式。

补充说明：

- 每组最少人数与最多人数，默认仅用于学生自发组队阶段的校验。
- 项目允许创建的最大小组数量，默认仅用于学生自发组队阶段的校验。
- 老师或助教在后续执行强制调组、合并、补位时，不受该人数限制约束。
- 老师在后续手动加组、删组、调组时，不受学生自组阶段的最大小组数限制约束。

### 场景 B：课程群与小组群

- 课程创建后自动生成课程群，用于班级公共通知与课程级讨论。
- 项目小组建立后自动生成小组群，用于组内沟通与协作留痕。
- 课程群用于公共信息，小组群用于执行协作，二者职责清晰分离。

### 场景 C：小组组建与协作空间开启(Project)

- 学生围绕课程项目组建小组。
- 系统为每个小组生成独立协作空间。
- 同一学生可在不同课程中参与不同小组，互不冲突。

组队面板建议采用表格化机制：

- 页面按组号展示当前项目下的可用小组位。
- 学生创建小组时，本质上是抢占一个当前可用组号。
- 已创建小组展示当前人数、组长、是否可加入等基础信息。
- 未占用组号显示为空位，便于学生直观看到剩余组队空间。

组队面板的学生操作：

- 创建小组：自动成为组长。
- 申请加入某组：由目标组长确认。
- 邀请同学入组：由被邀请人确认。
- 退出当前组：若为组长，需要先转让组长。
- 组长转让：需目标成员同意后生效。

即时校验规则：

- 是否为本班级成员。
- 是否已在当前项目的其他组。
- 目标组是否超员。
- 当前组号是否仍可被占用。
- 若失败，需要给出明确原因和下一步操作入口，例如提交变更申请。

这里的超员与人数限制，仅针对学生在组队面板中的自助操作。

组号规则建议如下：

- 组号在组队过程中按“先占先得”方式分配。
- 截止前允许出现中间空号，便于学生自由组队和调整。
- 超过组队截止期后，系统自动整理空缺组号并重新按顺序显示，形成第一版截止后分配结果。
- 在截止后，老师可以继续手动改组、加组、删组，用于完成教学管理上的最终调整。
- 老师完成最终确认后，系统再次自动按当前结果连续重排组号，生成最终分配结果。
- 重排后对外展示为连续组号，避免出现大量空洞编号。
- 内部应保留稳定的小组唯一标识，组号仅作为展示和管理编号使用。

老师/助教的直接管理权限：

- 可对具体学生执行调组。
- 可对多个小组执行合并。
- 可对未分组学生按人数和规则进行辅助分组。
- 原则上保留对人员归属的最高调整权，用于处理现实中的变动情况。
- 老师/助教执行强制安排时，可以突破学生自组阶段的人数上下限，以满足教学管理需要。

### 场景 D：小组任务协作（沿用原团队协作流程）(Task_MiniTask)

- 采用 Task-MiniTask 两层机制：
  - Task 由老师定义课程目标任务（例如“第一次汇报 PPT”）。
  - MiniTask 由小组拆分执行项（例如“文案”“图片”“演讲稿”）。
- 小组在 MiniTask 层完成分工、状态推进和协作落地。
- 除任务层级调整外，其余协作流程保持原有团队协作模式。

Task-MiniTask 职责边界：

- Task：定义“做什么、何时提交、验收标准”。
- MiniTask：定义“谁来做、怎么拆、当前进度”。

### 场景 E：小组沟通与过程留痕

- 组内保留项目沟通能力（聊天/讨论/文件共享）
- 沟通归属于课程下对应 project 小组，形成过程记录。

### 场景 F：小组变动与异常处理

- 小组允许发生人员变动，以适应真实教学场景中的临时调整。
- 当学生需要中途换组、加人、退组时，可以由老师或助教直接处理。
- 未满员小组可由老师或助教进行合并或补位。
- 所有组队变动、调组、合并操作都应保留过程记录。

截止后的处理建议：

- 截止后先对未占用组号进行自动清理。
- 截止后先对展示层组号进行连续化重排，生成第一版自动结果。
- 对未分组学生，由老师或助教决定是否补位、调组或单独处理。
- 对人数不满足要求的小组，由老师或助教执行合并、保留或手动调整。
- 若教学场景需要，老师或助教可保留超出常规人数限制的特殊小组。
- 老师完成所有手动调整后，再次触发自动连续化重排，生成最终组号分配。

### 场景 G：成果与材料提交

- 支持提交多种文件材料（报告、演示材料、压缩包等），结合老师的具体要求。
- 支持提交 skills 相关内容与其他补充文件。
- 仓库信息采用“提交字段”方式记录（如仓库链接或提交说明字段）。
- 文件管理仅做上传、归档、查阅，本阶段不做 RAG 检索与问答。

### 场景 H：批改与进度查阅

- 老师与助教查看小组提交内容和协作过程。
- 老师与助教进行批改、反馈与进度追踪。
- 教务处可查看课程层面的总体推进与批改覆盖情况。

### 场景 I：审计留痕与责任追踪（新增）

- 成绩登记、成绩修改、批改意见变更必须记录日志。
- 日志至少记录：操作人、角色、课程、对象、时间、变更前后值。
- 老师可查本课程日志；教务处可查课程级日志。

## 八、Task-MiniTask 机制细化（新增）

### 1. Task（老师层）

- 创建者：老师。
- 内容：目标描述、截止时间、提交要求、验收标准。
- 属性：课程级目标任务，强调统一要求与可评分性。

### 2. MiniTask（小组层）

- 创建者：组长或组员。
- 内容：执行项、负责人、优先级、状态、计划时间。
- 属性：小组内部执行与协同的最小单元。

### 3. 提案机制

- “成员提案任务-组长审核”工作流可以作为后续优化项。
- 当前版本不强制实现该机制，先保证 Task-MiniTask 主流程可用。

### 4. 建议的最小规则

- 每个 Task 可以直接使用，也可以继续拆分 MiniTask。
- Task 默认不可删除，可归档。
- MiniTask 可新增、编辑、关闭，用于真实协作推进。

## 九、需求清单（当前版本）

### A. 必要需求（当前必须落地）

- 课程管理：教务处创建课程、维护课程状态。
- 班级管理：课程下维护学生名单并限定组队范围。
- 项目发布：老师在课程下创建并发布项目。
- 组队配置：老师可设置组队时间、每组人数上下限、最大小组数、组号规则。
- 课程群与小组群：支持课程公共沟通与小组协作沟通。
- 小组机制：学生按课程项目组建小组。
- 组队面板：支持创建小组、申请入组、邀请入组、退出小组、组长转让，并展示表格式组号占用情况。
- 小组空间：每个小组有独立任务与协作空间。
- 多课程参与：同一学生可参与多门课程、多个项目小组。
- 团队协作：保留原有协作主流程（任务推进、组内沟通、过程记录）。
- 任务分层：支持 Task-MiniTask 两层结构。
- 提交管理：支持文件提交（与仓库字段提交）。
- 批改流程：老师/助教查阅、批改、反馈、查看进度。
- 助教子账号：支持老师创建助教子账号并绑定课程。
- 组队调整：老师/助教支持调组、合并、补位等强干预操作。
- 组号整理：组队截止后支持自动清理空号并按顺序重排展示组号。
- 管理视角：教务处查看课程级进度与执行情况。

### B. 有讨论余地的需求（先定方向，后定细节）

- Task 与阶段节点的绑定粒度。
- MiniTask 状态流转是否固定模板化。
- 提案机制何时上线（成员提案 -> 组长审核）。
- 助教是否允许直接发布阶段分数（默认不允许）。
- 审计留痕：成绩与批改相关关键操作必须记录并可查询。
- 组号重排是在截止后立即执行，还是由老师手动确认后执行。
- 未分组学生的自动补位规则细节。
- 小组人数上限、推荐人数和异常人数的处理策略。

### C. 暂不实现需求（明确后置）

- 文件管理相关 RAG（知识检索、语义问答、自动总结链路）。
- 复杂智能化自动评审闭环。

## 十、验收导向（仅针对场景与需求）

若以下问题都能回答为“是”，则本阶段场景与需求定义合格：

- 是否清楚区分了教务处、老师、助教、学生四类角色的职责？
- 是否明确了“课程 -> 班级 -> 项目 -> 小组 -> 协作与提交 -> 批改与查阅”的主链路？
- 是否定义了课程群、小组群和组队面板的基本操作？
- 是否明确了助教子账号的权限边界与责任归属？
- 是否落实了 Task-MiniTask 两层任务机制？
- 是否保留了老师/助教对组队变动的最高调整权？
- 是否能够支撑多课程、多小组并行参与的学生场景？
