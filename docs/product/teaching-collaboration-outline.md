# Linksee 教学项目协作与过程评价平台大纲

## 1. 项目定位

Linksee 是面向高校课程项目、小组作业、实训课程和毕业设计过程管理的云端协作平台。它不是单纯的线上提交系统，而是把小组项目协作、仓库链接与贡献说明、文件传输、阶段提交、教师反馈和评分依据统一起来，互评作为后续增强。

核心口号：

> 让学生协作有空间，让老师评价有依据。

不建议继续定位为通用团队协作平台，也不建议只做作业提交平台。通用协作市场已有飞书、钉钉、企业微信、Slack、Teams、Notion 等成熟产品；教学平台也已有线上提交和分组能力。Linksee 应该围绕“课程项目小组协作 + 教师实时追踪 + 过程评价”建立差异化。

## 2. 目标用户

### 2.1 老师

老师需要发布课程项目、设置阶段要求、追踪各小组进度、评价提交成果，并在期末给出有依据的成绩。

老师的核心诉求：

- 不想逐个翻群聊、网盘、邮件、代码仓库和压缩包。
- 想快速知道哪些组没交、哪些组延期、哪些组有风险。
- 想判断每个学生是否真实参与，包括任务、讨论、文件、仓库链接和贡献说明。
- 想把过程性评价和最终成绩结合起来。

### 2.2 学生

学生需要知道小组当前要完成什么、如何提交成果、老师给了什么反馈、自己需要补充什么材料。

学生的核心诉求：

- 小组有统一协作空间，不需要在多个工具之间来回切换。
- 提交流程清楚，不容易交错位置。
- 能看到阶段要求、截止时间和反馈。
- 能记录个人贡献，避免贡献被忽略。
- 能围绕小组任务协作推进。

### 2.3 助教

助教负责辅助老师检查提交物、验收说明和评分材料，互评作为后续增强。

助教的核心诉求：

- 快速筛选待检查、待评分、异常提交。
- 统一查看小组材料和历史反馈。
- 减少手工整理 Excel 和文件夹的时间。

### 2.4 学院或教务

学院或教务关心课程项目是否有过程记录、评分是否有依据、教学质量是否可追踪。

## 3. 典型痛点

1. 老师只能看到最终成果，看不见真实过程。
2. 小组材料分散在微信群、网盘、GitHub、学习通和邮件里。
3. 项目延期、跑偏、搭便车很难被早期发现。
4. 期末评分依赖老师印象和人工整理，公平性不足。
5. 阶段反馈没有形成闭环，学生是否修改难以追踪。
6. 非代码类项目很难使用 GitHub Classroom 这类工具管理。
7. 代码类项目又容易分散在 GitHub、群聊、网盘和课堂汇报中。
8. 验收说明、互评结果（后续增强）、文档提交、讨论记录和教师评分没有统一汇总。

## 4. 产品价值

### 4.1 对老师

- 一屏查看所有小组进度。
- 自动标记未提交、延期、待评价和风险小组。
- 通过 MiniTask、讨论、文件、仓库链接、贡献说明和阶段记录判断学生真实投入。
- 用 Rubric 规范评分，减少人工整理成本。

### 4.2 对学生

- 在一个小组空间内完成任务协作、聊天讨论、文件传输和阶段提交。
- 明确知道每个阶段交什么、什么时候交、交到哪里。
- 可以记录个人贡献和小组分工。
- 能及时看到老师反馈并迭代成果。

### 4.3 对学校

- 支持过程性评价改革。
- 沉淀课程项目全过程数据。
- 提升实训、课程设计、毕业设计中期检查的管理效率。

## 5. 核心产品结构

### 5.0 术语边界

为降低开发歧义，产品和代码统一使用：

- Course：课程空间。
- Assignment：老师发布的课程项目。
- Stage：Assignment 下的阶段要求，也承担老师层任务职责。
- Group：学生项目小组。
- MiniTask：小组内部拆分的执行任务。
- Submission：小组针对 Stage 的成果提交。
- Review / Grade：反馈与成绩。

P0 不单独实现老师层 `Task`，避免和 `Stage` 重复。学生端任务看板使用 `MiniTask`。

### 5.1 老师端

老师端强调“实时看见协作过程”和“评价”。

- 课程首页：课程项目、班级、小组、阶段进度。
- 班级追踪看板：所有小组状态汇总。
- 待处理中心：待评价、待复核、待测试、待反馈。
- 小组详情页：MiniTask、提交物、讨论、文件、仓库链接、贡献说明、评分、反馈。
- 评分台：Rubric、验收说明、最终成绩（互评为后续增强）。

### 5.2 学生端

学生端强调“真实协作”和“简单提交”。

- 我的课程：当前课程项目列表。
- 我的小组：成员、分工、当前 Stage 与 MiniTask。
- 小组协作空间：MiniTask 看板、聊天讨论、文件传输、代码仓库链接。
- 提交台：上传文件、填写说明、提交链接、记录贡献。
- 反馈页：老师反馈、修改要求、评分结果。
- 小组任务板：待办、进行中、已完成。

### 5.3 助教端

助教端强调“检查效率”。

- 待检查提交列表。
- 异常提交筛选。
- 验收说明核查。
- 互评异常提醒（后续增强）。
- 评分材料汇总。

## 6. MVP 功能清单

MVP 范围以 [docs/product/mvp-p0-and-follow-up-tasks.md](docs/product/mvp-p0-and-follow-up-tasks.md) 为准，本节为概览说明。

第一版不做大而全，重点完成“老师发布阶段要求 -> 学生小组协作 -> 学生提交 -> 老师追踪和评价”的闭环。

### 6.1 课程与角色

- 创建课程
- 创建 Assignment（课程项目）
- 导入学生名单
- 创建小组
- 设置老师、助教、学生角色

### 6.2 阶段里程碑

- 阶段名称：选题、需求、设计、实现、测试、答辩
- 阶段说明
- 截止时间
- 提交要求
- 评分权重

### 6.3 学生提交

- 上传文件：PDF、PPT、ZIP、图片、视频
- 提交链接：GitHub、GitLab、原型链接、网盘链接
- 填写提交说明
- 填写本阶段个人贡献
- 查看提交状态

### 6.4 小组协作

- 小组任务看板
- 小组聊天讨论
- 文件传输与材料归档
- 代码仓库链接
- 小组动态时间线
- 老师可查看但不打断学生协作
- 老师可在关键节点评论或反馈

### 6.5 老师评价

- 查看提交物
- 设置状态：需修改、已通过；不通过原因写入反馈说明，不单独设置 `rejected` 状态
- 填写反馈
- Rubric 打分
- 生成阶段评分

### 6.6 追踪看板

- 小组进度总览
- 未提交小组
- 已提交待评价小组
- 延期小组
- 贡献异常小组
- 讨论活跃度
- 文件提交记录
- 仓库链接与人工贡献记录
- 阶段完成率

## 7. 后续增强功能

### 7.1 互评

- 小组成员互评
- 贡献比例填写
- 异常互评提醒
- 互评结果进入评分参考

### 7.2 自动测试

- 老师配置测试说明或测试脚本。
- 学生提交软件包或仓库链接。
- 系统记录测试结果。
- 测试通过率进入评分参考。

### 7.3 GitHub Classroom 集成

Linksee 不需要替代 GitHub Classroom，而是可以补齐它之外的小组协作和教学评价层。

- 关联 GitHub 仓库
- 汇总 commit 记录
- 展示 GitHub Actions 测试结果
- 将自动代码贡献统计作为过程证据之一
- 将代码仓库、文件、讨论、互评和教师评分汇总到同一小组档案

说明：自动 commit 汇总、代码贡献统计和 GitHub Actions 结果展示属于后续增强。MVP 只要求记录仓库链接和学生填写的贡献说明。

### 7.4 AI 助教

- 自动生成小组周报。
- 自动总结提交材料。
- 自动识别长期无进展、延期、贡献不均等风险。
- 生成教师评语草稿。
- 基于课程资料和提交材料进行问答。

### 7.5 教学数据分析

- 各阶段提交率
- 小组活跃度
- 个人贡献趋势
- 评分分布
- 课程项目质量复盘

## 8. 与竞品的区别

| 工具 | 主要能力 | Linksee 的差异 |
| --- | --- | --- |
| 飞书/钉钉 | 沟通、文档、组织协作 | 不以课程项目、评分和过程评价为核心 |
| 学习通/雨课堂 | 课程、作业、测验 | 对小组内部协作、贡献说明和过程追踪较弱 |
| GitHub Classroom | 代码作业、仓库、自动测试 | 不覆盖聊天协作、文件传输、非代码材料、Rubric、互评（后续增强）和班级过程看板 |
| Jira/Trello | 任务管理 | 老师评分、教学反馈、课程角色不是核心 |
| Excel/网盘 | 灵活但分散 | 缺少自动追踪、统一评价和证据汇总 |

一句话区分：

> GitHub Classroom 看代码提交，Linksee 看小组项目如何协作完成，并把过程变成老师可评价的依据。

## 9. UI 设计方向

Linksee 的 UI 竞争力应该来自“协作入口简单、老师视角清楚、低学习成本”。

### 9.1 学生端入口

学生不需要看到复杂项目管理概念，只需要三个入口：

- 本周要交什么
- 进入小组空间
- 上传成果或代码链接
- 查看老师反馈

### 9.2 老师端入口

老师不需要逐个进入小组翻材料，只需要先看到：

- 哪些组没交
- 哪些组待评价
- 哪些组延期
- 哪些组贡献异常
- 哪些组协作不活跃
- 哪些组需要我反馈

### 9.3 评分端入口

评分界面要接近老师熟悉的表格和 Rubric：

- 评分项
- 权重
- 分数
- 评语
- 过程证据
- 一键导出成绩表

## 10. 商业说服点

如果甲方是学校、学院或课程负责人，可以这样说明：

> 现在老师管理小组项目时，常常要同时查看群聊、网盘、GitHub、学习通和 Excel。Linksee 把小组协作、仓库链接、贡献说明、文件提交、阶段反馈、互评（后续增强）和评分统一到一个平台，帮助老师减少材料整理时间，并让过程性评价更公平、更有依据。

重点收益：

1. 降低老师管理成本。
2. 提高小组作业评分公平性。
3. 支持过程性评价改革。
4. 及时发现项目延期和搭便车问题。
5. 同时适配代码类和非代码类课程项目。
6. 让学生在课程项目中形成真实协作记录，而不是只在期末上传成果。

最短销售话术：

> Linksee 不是再做一个提交入口，而是让课程小组项目从协作到评分都有迹可循。

## 11. 建议演示脚本

1. 老师创建“软件工程课程设计”项目。
2. 设置五个阶段：选题、需求、设计、实现、测试。
3. 导入 6 个小组。
4. 学生在“提交台”上传需求文档和原型链接。
5. 学生在小组空间里拆任务、讨论、上传文件并关联 GitHub 仓库。
6. 老师打开“班级追踪看板”，看到 2 组未提交、1 组延期、1 组协作不活跃。
7. 老师进入某小组详情，查看提交材料、仓库链接、贡献说明、文件记录和讨论。
8. 老师按 Rubric 打分并给出反馈。
9. 系统生成阶段评价摘要和待改进事项。

## 12. 与当前工程骨架的映射

当前项目已有的团队协作骨架可以保留，但需要换成教学语义。

| 原概念 | 教学版概念 |
| --- | --- |
| Team | Course / Class |
| Project | Assignment / Course Project |
| Task | Stage / MiniTask |
| Chat | Group Discussion / Teacher Feedback |
| Docs | Submissions / Course Materials |
| Feed | Progress Timeline |
| RAG | Course Knowledge Q&A |
| AI 站会助手 | AI Teaching Assistant |

建议第一条端到端链路：

> 老师发布阶段要求 -> 小组在线协作 -> 小组提交成果 -> Socket 推送待评价通知 -> 老师查看协作过程并评分反馈 -> 学生收到反馈并修改。
