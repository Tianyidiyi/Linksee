# Linksee 两个月开发分工与进度计划

## 1. 目标

两个月内完成一个可演示、可答辩的 Linksee MVP：

> 老师发布阶段要求 -> 学生小组在线协作 -> 小组提交成果 -> 老师实时查看过程 -> 老师评分反馈。

第一版不要追求完整教学平台，而是聚焦“教学项目小组协作 + 老师过程追踪 + 阶段评价”这条主线。

## 2. MVP 范围

### 2.1 必做功能

1. 用户登录与角色
   - 老师
   - 学生
   - 助教账号与权限模型保留；若时间紧，助教专属页面可延后

2. 课程与小组
   - 老师创建课程
   - 老师创建 Assignment（课程项目）
   - 创建小组
   - 学生加入小组

3. 小组协作空间
   - 小组任务看板
   - 小组讨论
   - 文件上传或文件记录
   - GitHub 仓库链接记录

4. Stage 阶段要求与提交
   - 老师设置阶段里程碑
   - 学生提交阶段成果
   - 提交状态：未提交、已提交、需修改、已通过

5. 老师追踪看板
   - 查看所有小组进度
   - 查看未提交、延期、待评价小组
   - 查看小组任务、讨论、文件、提交记录

6. 评分与反馈
   - 老师填写反馈
   - 简化 Rubric 打分
   - 查看每组阶段评分

### 2.2 可选增强

时间充足再做：

- 组内互评
- 贡献统计图表
- GitHub commit 记录导入
- AI 周报/评语草稿
- 自动测试结果接入
- 成绩导出 Excel

### 2.3 明确不做

两个月 MVP 不建议做：

- 完整学校教务系统
- 复杂审批流
- 企业级复杂权限配置后台
- 真正在线代码编辑器
- 完整 GitHub Classroom 替代能力
- 私有化部署和多租户商业化能力

## 3. 五人分工

建议采用“角色负责制 + 每周集成”的方式。每个人有主责，但不要完全割裂，关键接口和页面需要一起对齐。

## 3.1 A：项目负责人 / 产品与集成

主责：

- 统一产品范围，防止功能膨胀。
- 维护需求文档、页面流程和演示脚本。
- 组织每周例会和验收。
- 协调前后端接口。
- 最后负责答辩材料、PPT 和演示路径。

交付物：

- 产品功能清单
- 页面流程图
- 每周进度记录
- 最终演示脚本
- 答辩 PPT 大纲

适合人选：

- 沟通能力强
- 对整体产品比较敏感
- 能催进度，也能做取舍

## 3.2 B：后端负责人

主责：

- 搭建 API 服务。
- 设计数据库模型。
- 实现登录、课程、小组、任务、提交、评分等接口。
- 维护接口文档。
- 和前端对接数据格式。

核心模块：

- Auth
- Course
- Group
- Assignment
- Stage
- Submission
- Review / Grade

交付物：

- 后端可运行服务
- 数据库表结构
- REST API
- 基础接口测试
- Demo 数据初始化脚本

## 3.3 C：前端负责人

主责：

- 搭建前端页面框架。
- 实现老师端和学生端主要页面。
- 负责页面交互和状态管理。
- 和后端联调接口。

核心页面：

- 登录页
- 学生首页
- 小组协作空间
- 阶段提交页
- 老师课程看板
- 小组详情页
- 评分反馈页

交付物：

- 可运行前端应用
- 核心页面
- API 调用封装
- 页面联调结果

## 3.4 D：实时协作与文件模块负责人

主责：

- 基于现有 Socket 骨架实现实时通知。
- 实现小组讨论或消息模块。
- 实现文件上传/文件记录。
- 实现老师端实时看到提交或讨论更新。

核心模块：

- 小组讨论
- 实时通知
- 文件提交记录
- GitHub 链接记录
- 小组动态时间线

交付物：

- Socket 事件定义
- 讨论消息接口或实时消息
- 文件上传/文件元数据接口
- 小组动态流

## 3.5 E：测试、数据、评分与展示负责人

主责：

- 设计测试用例。
- 准备演示数据。
- 实现或协助实现评分 Rubric、贡献记录、统计展示。
- 负责最终验收和问题清单。

核心模块：

- Rubric 评分
- 贡献记录
- 统计卡片
- 测试用例
- 演示数据

交付物：

- 测试清单
- Bug 列表
- 演示课程数据
- 评分与贡献展示
- 最终验收报告

## 4. 八周进度安排

## 第 1 周：定范围与搭骨架

目标：

- 全组统一产品方向。
- 跑通项目基础环境。
- 确认第一版页面和接口范围。

任务：

- A：整理最终 MVP 功能清单和演示路径。
- B：确定数据库模型草案。
- C：确定页面结构和路由。
- D：梳理 Socket/文件/讨论模块的最小方案。
- E：设计测试清单和演示数据需求。

验收标准：

- 全组确认不再做泛团队协作平台。
- 形成页面清单、接口清单、数据模型草案。
- 本地项目能启动基础 demo 或空页面。

## 第 2 周：基础账号、课程、小组

目标：

- 跑通用户、课程、小组的基本链路。

任务：

- B：实现登录、课程、小组相关 API。
- C：实现登录页、课程首页、小组列表页。
- A：补充课程和小组用户流程。
- D：预留小组空间入口。
- E：准备老师、学生、小组的 demo 数据。

验收标准：

- 老师能登录并创建课程。
- 老师能创建小组。
- 学生能看到自己所在小组。

## 第 3 周：Stage 阶段要求与学生提交

目标：

- 跑通老师发布阶段要求、学生提交成果。

任务：

- B：实现 Assignment、Stage、Submission API。
- C：实现阶段列表、提交页。
- D：实现文件上传或文件记录能力。
- A：确认提交状态流转。
- E：补充提交相关测试用例。

验收标准：

- 老师能创建 Stage 阶段要求。
- 学生能上传或登记提交物。
- 提交状态能正确展示。

## 第 4 周：小组协作空间

目标：

- 让学生真的能在系统中协作，而不是只提交。

任务：

- C：实现小组协作空间页面。
- D：实现小组讨论、任务看板、动态时间线。
- B：提供任务、讨论、动态相关接口。
- A：确认学生端体验是否足够简单。
- E：准备多个小组协作状态的演示数据。

验收标准：

- 小组能创建任务、更新任务状态。
- 小组能进行讨论。
- 小组详情能看到动态记录。

## 第 5 周：老师追踪看板

目标：

- 老师能实时看到每组进度和风险。

任务：

- C：实现老师课程看板、小组详情页。
- B：实现看板汇总接口。
- D：实现提交/讨论/任务更新的实时通知或刷新机制。
- A：定义风险规则，例如未提交、延期、无活动。
- E：验证看板数据准确性。

验收标准：

- 老师能看到所有小组进度。
- 老师能筛选未提交、延期、待评价小组。
- 老师能进入小组详情查看任务、讨论、文件、提交。

## 第 6 周：评分反馈与贡献记录

目标：

- 跑通老师评价、学生查看反馈。

任务：

- B：实现 Review、Grade、Contribution API。
- C：实现评分反馈页、学生反馈页。
- E：设计简化 Rubric 和贡献记录展示。
- A：确认评分规则适合答辩说明。
- D：补充评分通知或反馈动态。

验收标准：

- 老师能给阶段提交打分和写反馈。
- 学生能看到反馈和状态。
- 小组详情能展示贡献记录。

## 第 7 周：联调、测试、补短板

目标：

- 把所有核心链路连起来，进入可演示状态。

任务：

- 全组：集中联调。
- B：修复接口和数据问题。
- C：修复页面问题。
- D：修复实时、讨论、文件问题。
- E：执行测试清单并维护 Bug 列表。
- A：准备演示脚本和答辩材料。

验收标准：

- 完整链路可跑通。
- 主要页面无明显阻塞 Bug。
- 演示数据完整。

## 第 8 周：打磨、部署、答辩准备

目标：

- 完成最终演示版本。

任务：

- A：完成 PPT、产品说明、演示讲稿。
- B：整理 API 和部署说明。
- C：统一 UI 细节和空状态。
- D：确保实时/文件/讨论功能演示稳定。
- E：完成最终测试报告和已知问题说明。

验收标准：

- 可稳定演示 10 到 15 分钟。
- 有一套完整 demo 数据。
- 有 README、产品大纲、部署说明、测试报告。
- 答辩时能清楚说明竞品差异和商业/教学价值。

## 5. 每周例会流程

建议每周至少两次同步：

### 周初计划会

时间：30 到 45 分钟。

每个人回答：

1. 本周要交付什么？
2. 依赖谁？
3. 最大风险是什么？

### 周末验收会

时间：45 到 60 分钟。

每个人演示：

1. 已完成内容。
2. 未完成原因。
3. 下周需要谁配合。

A 负责记录结论，避免口头同步后没人跟进。

## 6. 推荐功能优先级

### P0：必须完成

- 登录与角色
- 课程/小组
- Stage 阶段要求
- 学生提交
- 小组任务看板
- 小组讨论
- 老师追踪看板
- 评分反馈

### P1：尽量完成

- 文件上传
- 小组动态时间线
- 贡献记录
- 简化 Rubric
- GitHub 仓库链接
- 实时通知

### P2：有时间再做

- GitHub commit 导入
- 自动测试结果展示
- 组内互评
- AI 周报
- 评分导出

## 7. 推荐演示路径

最终答辩建议按这个故事演示：

1. 老师登录，进入“软件工程课程设计”。
2. 老师查看班级中 5 个小组的项目进度。
3. 老师发布“需求分析阶段”任务。
4. 学生登录，进入自己的小组空间。
5. 学生创建任务、聊天讨论、上传文件、填写 GitHub 链接。
6. 学生提交阶段成果。
7. 老师看板实时出现“待评价”状态。
8. 老师进入小组详情，查看任务、讨论、文件、提交、贡献。
9. 老师按 Rubric 打分并写反馈。
10. 学生看到反馈并修改提交。

## 8. 风险与控制

### 8.1 功能膨胀

风险：

- 想做 AI、自动测试、GitHub 深度集成、复杂数据分析，最后核心链路没完成。

控制：

- 第 6 周前只做 P0 和少量 P1。
- P2 功能必须等核心链路稳定后再做。

### 8.2 前后端联调太晚

风险：

- 第 7 周才发现接口和页面对不上。

控制：

- 第 2 周开始就用 mock 数据和真实接口并行联调。
- 每周至少合并一次可运行版本。

### 8.3 UI 太复杂

风险：

- 做成 Jira/飞书式复杂协作工具，老师和学生不好理解。

控制：

- 学生端只保留“进入小组空间、查看本周任务、上传成果、查看反馈”。
- 老师端只保留“看进度、看风险、看提交、给反馈”。

### 8.4 评分逻辑太复杂

风险：

- Rubric、互评、贡献统计做得太细，影响主线。

控制：

- 第一版只做简化 Rubric：评分项、权重、分数、评语。
- 贡献记录先人工填写，后续再自动统计。

## 9. 两个月后的理想成果

两个月结束时，Linksee 应该能清楚展示：

- 学生确实能在平台内协作推进小组项目。
- 老师确实能实时查看小组过程。
- 平台能沉淀任务、讨论、文件、提交、贡献和评分记录。
- 最终评分不只看结果，而是有过程依据。
- Linksee 和学习通、飞书、GitHub Classroom 有清晰差异。
