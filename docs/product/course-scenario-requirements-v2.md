# 课程场景需求说明 v2（面向当前后端实现）

> 更新时间：2026-05-11  
> 本文聚焦课程主链路与组队规则，和后端现状对齐。

## 1. 场景目标
- 支持“课程 -> 作业 -> 阶段 -> 小组 -> 提交/评审”的教学协作主流程。
- 保证学生组队可自助、教学侧可兜底管理。
- 前后端对齐最小可用规则，优先可落地。

## 2. 角色与权限（课程域）
- 教务（academic）：全局课程管理与教师/助教关系管理。
- 教师（teacher）：管理自己课程下的作业、阶段、小组与评审。
- 助教（assistant）：在绑定课程内协助管理（受课程绑定约束）。
- 学生（student）：参与课程、组队、自助申请入组、提交作业。

## 3. 组队 v1 简化需求（已采用）

### 3.1 学生只做两件事
- 创建小组（自动成为组长）。
- 申请加入已有小组（由组长审批）。

### 3.2 组长只做三件事
- 同意/拒绝入组申请。
- 踢出组员。
- 发起组长转让（需目标成员确认）。

### 3.3 截止规则
- 学生侧组队操作在 `groupFormEnd` 后关闭。
- 教师/助教仍可做手动调整（教学兜底）。

### 3.4 暂不纳入 v1 的复杂能力
- 邀请入组流程。
- 自动重排组号（截止前后两轮）。
- 批量合并组、智能补位、超员特批流程入口。
- 复杂退组链路（组长先转让再退出的自动引导）。

## 4. 关键业务规则
- 学生必须是课程有效成员，才能参与组队。
- 同一学生在同一作业下只能属于一个组。
- 学生自助入组受 `groupMaxSize` 限制。
- 同一小组同一时刻只允许一条待处理组长转让请求。

## 5. 后端现状对齐（本版）

### 5.1 已实现
- 课程/教师/助教/成员管理主接口。
- 作业与阶段 CRUD、状态控制、附件上传（MinIO）。
- 小组列表/创建、成员手动增删。
- 入组申请链路：申请、查询、同意、拒绝。
- 组长转让链路：发起、接受、拒绝（双向确认）。

### 5.2 未实现
- 组号自动重排与截止后自动整理。
- 教学侧批量调组、合并组、智能补位专用接口。
- 组队异常工单化流程（例如超员特批流程入口）。

## 6. 前端对齐建议（简版）
- 组队面板字段先固定为：组号、组长、当前人数、是否可申请加入。
- 学生端按截止状态动态禁用“创建/申请”按钮。
- 组长端增加“申请列表”与“转让组长待确认”卡片。
- 教师/助教端保留手动加减成员入口，作为截止后兜底操作。
