# 数据库迁移与数据安全规范

本文档用于约束 MySQL 在迭代中的结构变更，目标是确保“数据库可演进，历史数据不丢失”。

## 1. 基本原则

1. 所有结构变更必须通过 migration 执行，禁止手工改线上库。
2. 迁移脚本必须可重复验证，并记录在版本控制中。
3. 迁移前必须备份，迁移后必须校验。
4. 任何破坏性变更（删字段、改类型）必须经过兼容期。

## 2. 推荐迁移策略：Expand -> Migrate -> Contract

1. Expand（扩展）
- 新增字段/新表/新索引
- 旧代码仍可运行

2. Migrate（迁移）
- 通过脚本回填数据
- 双写或灰度切换到新字段

3. Contract（收缩）
- 在验证稳定后删除旧字段/旧索引
- 删除前保留至少一个发布周期的观察期

## 3. 每次迁移前检查清单

1. 变更说明：为什么改、影响哪些模块。
2. 风险评估：是否包含锁表风险、大字段重写风险。
3. 回滚方案：迁移失败如何恢复。
4. 备份执行：生成可恢复备份文件。
5. 变更窗口：避免高峰时段操作。

## 4. 每次迁移后校验清单

1. 核心表行数是否异常变化。
2. 新增字段是否按预期填充。
3. 关键接口（登录、任务流转、文件上传）是否通过冒烟测试。
4. 慢 SQL 与错误日志是否显著增加。

## 5. Docker 环境下的数据保护要点

1. 使用命名卷保存 MySQL 数据（当前为 mysql_data）。
2. 日常停止请使用 down，不要使用 down -v。
3. 只有在明确要清空数据时才允许 down -v。
4. 备份文件必须存放在仓库外或独立备份目录。

## 6. 团队协作约定

1. 每个 migration 对应一个需求或缺陷编号。
2. Pull Request 必须包含：迁移脚本 + 风险说明 + 回滚说明。
3. 主分支发布前，必须在测试环境完成一次完整迁移演练。

## 7. 建议发布流程

1. 备份数据库。
2. 执行 migration。
3. 执行数据回填脚本（如有）。
4. 运行冒烟测试。
5. 观察 30 分钟关键监控。
6. 如异常，立即执行回滚方案。
