# 通信架构规划（MVP 到可扩展）

本文基于当前项目既有设计：

- MVP：任务管理 + 即时通讯 + 文档中心
- 增强：RAG 与 AI 助手（可后置）
- 技术栈：Vue3 + TypeScript，Node.js + Express/NestJS，MySQL + Prisma，Redis + Socket.io

## 1. 一句话结论

当前项目建议采用混合通信模型：

- HTTP：业务主链路与 CRUD
- Socket：即时消息、在线状态、通知推送
- 事件/队列：异步任务（文档处理、通知汇总、后续 RAG 索引）

这样可以在 MVP 阶段快速交付，同时给后续扩展留出清晰边界。

## 2. 先做什么，不做什么

MVP 必做：

1. 用户认证与 RBAC
2. 任务/项目的 HTTP API
3. 聊天消息的 Socket 收发
4. 关键通知推送（任务指派、评论、提及）

MVP 可延后：

1. 文件内容向量化与检索
2. RAG 问答
3. 复杂事件总线编排（先用轻量队列）

说明：RAG 不必立即实现，但必须预留异步处理链路和数据模型扩展位。

## 3. 通信分层设计

### 3.1 HTTP（同步）

适用：

- 登录、鉴权、团队管理
- 项目/任务 CRUD
- 历史消息分页查询
- 文件元数据管理

必须约定：

- 接口超时：3 到 8 秒
- 幂等：写操作支持幂等键（如创建评论、重复提交）
- 错误码：未认证、无权限、资源不存在、冲突

### 3.2 Socket（实时）

适用：

- 新消息推送
- 在线状态变化
- 输入中状态（typing）
- 已读回执（可先做简化）

必须约定：

- 连接鉴权：JWT 握手校验
- 房间模型：team:{id}、project:{id}、channel:{id}
- 心跳保活：服务端 ping/pong
- 断线重连：客户端指数退避

### 3.3 事件与异步（worker）

适用：

- 文件上传后处理
- 通知汇总下发
- 后续 RAG 切片/向量化

建议事件示例：

- doc.uploaded
- doc.process.requested
- doc.process.completed
- task.assigned
- chat.message.created

## 4. RAG 后续预留（不强行现在实现）

建议先留三个能力接口：

1. 文档处理入口
- 输入：文件 ID
- 输出：处理状态（pending/processing/done/failed）

2. 检索接口占位
- 输入：query、scope
- 输出：空结果或 mock 结构（保持协议稳定）

3. 问答接口占位
- 输入：question、context scope
- 输出：暂时由普通搜索或提示文案替代

这样未来接入向量库时，前后端协议不用大改。

## 5. 你后续可能遇到的扩展问题

### 5.1 是否需要上消息队列

触发条件：

- Socket 事件明显增多
- 通知和文档处理互相阻塞
- worker 任务失败重试需求增加

建议：

- MVP 先用 Redis 队列或轻量任务队列
- 用户规模增长后再引入更完整的消息中间件

### 5.2 是否需要多实例 Socket

触发条件：

- 单实例连接数压力大
- 需要横向扩容

建议：

- 预留 Redis Adapter
- 会话与房间状态不要只放进程内存

### 5.3 数据一致性风险

常见问题：

- HTTP 写成功但 Socket 推送失败
- 异步任务重复执行

建议：

- 采用 outbox 思路或可靠事件记录
- 每个事件带唯一业务 ID，消费端去重

## 6. 新手可直接使用的“向 AI 提需求”模板

请把需求写成下面结构：

1. 目标
- 我要实现什么功能（例如：任务评论 + 实时提醒）

2. 现状
- 我已有哪些模块（auth、task、chat）
- 技术栈是什么

3. 业务时效
- 哪些需要实时（秒级）
- 哪些允许延迟（分钟级）

4. 约束
- 开发周期
- 团队经验
- 成本优先或性能优先

5. 期望输出
- 通信方案结论（HTTP/Socket/异步分别做什么）
- 接口草案
- 风险点与回滚方案
- 本周可执行任务清单

## 7. 示例提问（可直接复制）

请基于当前项目架构，设计“任务评论 + 实时通知”的通信方案。

已知：

- 前端 Vue3 + TS，后端 Node.js，数据库 MySQL + Prisma，实时计划用 Socket.io
- 目前有 auth、project-task、chat、docs、feed、worker
- RAG 暂不实现，但要保留后续接入能力

请输出：

1. 这个需求里 HTTP、Socket、异步任务分别承担什么
2. 需要新增的 6 到 10 个事件定义（名字、触发时机、关键字段）
3. 鉴权和权限校验放在哪一层
4. 失败重试与降级策略
5. 按后端/前端/测试拆分的一周开发计划

## 8. 推荐的当前决策

建议现在就确定：

1. MVP 使用 HTTP + Socket + 轻量 worker 三层通信
2. RAG 只做接口与任务链路预留，不做完整实现
3. 所有新增功能先判定实时等级，再决定通信方式

如果后续你确认用户规模较小（例如课程项目阶段），这套方案已经足够稳且可控。
